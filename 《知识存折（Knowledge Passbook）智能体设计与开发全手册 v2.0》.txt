# 《知识存折（Knowledge Passbook）智能体设计与开发全手册 v2.0》

# 第一部分：产品核心定义 (Product Definition)

## 1. 产品哲学
*   **核心理念**：知识不是单纯的记录，而是**资产的铸造**。
*   **解决痛点**：
    1.  **虚假勤奋**：只收藏不内化。
    2.  **断点焦虑**：学习被打断后难以接续。
    3.  **孤岛效应**：知识点之间互不关联，无法产生复利。
*   **关键差异点**：引入**“异步持久化”**机制。用户不需要一次性完成所有步骤，系统允许“半成品”存在，并负责在合适的时机唤醒它。

## 2. 核心隐喻
*   **存折 (Passbook)**：用户的知识总账。
*   **铸币 (Minting)**：将模糊的思维打磨成确定性资产的过程。
*   **孵化器 (Incubator)**：存放“半成品”和“草稿”的安全区。

# 第二部分：业务流程与交互逻辑 (User Journey)

系统采用**“30分钟异步循环 (The Asynchronous 30-Min Loop)”**：

### 阶段 1：存入专注 (Commit)
*   **动作**：用户输入“我要学什么”。
*   **系统**：
    *   生成任务 ID，状态标记为 `Creating`。
    *   启动 30 分钟倒计时（支持暂停，处理突发干扰）。
    *   **环境锁定**：界面进入“沉浸模式”，隐藏其他干扰信息。

### 阶段 2：收割与孵化 (Harvest & Incubate)
*   **触发**：时间到。
*   **动作**：弹出“5问模板”。
*   **分支逻辑（核心优化）**：
    *   **A. 顺畅完成**：用户填完，点击“提交预审”。 -> 进入 **阶段 3**。
    *   **B. 还没想好**：用户填写了部分，或者点击“存入孵化器”。 -> 状态转为 `Incubating`，任务挂起。
    *   **C. 意外中断**：浏览器关闭。 -> 系统自动保存草稿至 `Incubating`。

### 阶段 3：打磨与教练 (Refine & Coach)
*   **场景**：用户针对 v1.0 进行修改，或从孵化器唤醒旧任务。
*   **AI 角色**：**认知教练 (Cognitive Coach)**。
*   **交互**：
    *   不再直接打分拒绝。
    *   **旁注式反馈**：AI 在用户文本旁生成气泡——“这里提到了‘底层逻辑’，能具体举个例子吗？”
    *   **比对视图**：左侧显示 v(n-1) 的痛点，右侧编辑 v(n)。

### 阶段 4：签发入库 (Minting)
*   **权利**：**用户主权**。
*   **动作**：用户无视或采纳 AI 建议后，点击**【盖章入库】**按钮。
*   **系统**：
    1.  生成精美的“洞见卡片”图片。
    2.  写入向量数据库，触发关联分析。
    3.  状态转为 `Minted`（已入库）。

# 第三部分：功能规格与模块详解 (Functional Specs)

## A. 智能体工作台 (The Agent Workbench)

这是用户的主要操作界面，需适配桌面与移动端。

1.  **专注计时器 (Focus Timer)**
    *   **功能**：倒计时、暂停/继续、白噪音（可选）。
    *   **韧性设计**：异常退出后，重新打开网页自动恢复倒计时状态。

2.  **双态编辑器 (Dual-State Editor)**
    *   **v1.0 录入模式**：单纯的表单，5个输入框（是什么、怎么理解、用在哪、第一动作、为什么）。
    *   **vN.0 迭代模式**：
        *   **桌面端**：左右分栏。左侧是“历史版本+AI建议”，右侧是“当前编辑”。
        *   **移动端**：上下堆叠，支持手势滑动切换“看建议”和“写内容”。

3.  **孵化器看板 (The Incubator)**
    *   **展示**：卡片列表，显示“挂起时长”（如：已孵化 12 小时）。
    *   **唤醒机制**：每日首次登录，Agent 提示：“你还有 2 个关于‘Python并发’的草稿，今天要继续打磨吗？”

## B. 资产视图 (Asset Views)

1.  **存折明细（列表视图）**
    *   **分类**：按场景标签（决策、沟通、技能、心态）。
    *   **排序**：按“复利值”（被关联引用的次数）排序。

2.  **局部关联图（上下文视图）**
    *   **逻辑**：不默认展示全景。
    *   **交互**：点击任意一张卡片，下方展开“星系图”——只显示该卡片及与其**强相关**（相似度 > 0.85 或 手动连接）的 3-5 个邻居。

# 第四部分：AI Prompt 核心设计 (Agent Design)

我们将 AI 从“严厉的考官”重塑为“苏格拉底式的教练”。

### 1. 认知教练 Prompt：【启发式打磨】
*   **Trigger**：用户提交 v1.0 或 v2.0 时。
*   **System Prompt**：
    > 你是用户的认知教练。你的目标不是评判对错，而是通过提问逼出用户的深度思考。
    > **原则：**
    > 1. **不要**直接修改用户的文字。
    > 2. **要**针对“虚词”（如：优化、赋能、底层逻辑）进行追问：“这具体是指什么动作？”
    > 3. **要**检查“第一动作”的可执行性。如果用户写“提高沟通技巧”，你要问：“下次开会时，你会说的第一句话是什么？”
    > **输出格式**：JSON，包含 `highlight_text` (高亮原文) 和 `coach_suggestion` (教练建议)。

### 2. 关联分析师 Prompt：【惊喜连接】
*   **Trigger**：资产【盖章入库】成功后。
*   **System Prompt**：
    > 你是知识复利分析师。用户刚存入了一笔新资产 A。请在向量库中搜索最相似的旧资产 B。
    > **任务**：
    > 1. 如果 A 和 B 相似度 > 0.9：提示“重复资产，建议合并”。
    > 2. 如果 A 和 B 相似度在 0.7-0.9 且领域不同（如编程 vs 烹饪）：提示“发现跨界灵感！这是否意味着 [编程逻辑] 和 [烹饪技巧] 有共同之处？”
    > **语气**：兴奋、发现新大陆的感觉。

## 3. 状态机逻辑 (LangGraph Pseudocode)

```python
# 定义图的状态
class AgentState(TypedDict):
    content: dict
    feedback: str
    status: str # 'draft', 'reviewing', 'minted'

# 节点：教练反馈
def coach_node(state):
    suggestion = llm_coach.invoke(state['content'])
    return {"feedback": suggestion, "status": "reviewing"}

# 节点：入库处理
def mint_node(state):
    vector_db.upsert(embed(state['content']))
    return {"status": "minted"}

# 构建图
workflow = StateGraph(AgentState)
workflow.add_node("incubator", lambda x: x) # 停留节点
workflow.add_node("coach", coach_node)
workflow.add_node("mint", mint_node)

# 边逻辑：用户决定
workflow.add_conditional_edges(
    "coach",
    lambda x: "mint" if x['user_action'] == 'approve' else "incubator"
)
```

✅ 推荐流程：隐式版本迭代 (Implicit Versioning)
我们不要让用户手动去点“保存 v1.0”，那样太繁琐。我们要把这个过程做成自动化的。
核心流程设计：
草稿期 (Drafting)：
用户在 5 个框里输入内容。此时是草稿状态。
里程碑 v1.0 (The Audit Trigger)：
动作：用户点击**【提交给 AI 教练审计】**。
系统后台：
立刻将当前的文本保存为 Version 1（存入 asset_versions 表）。
AI 分析 v1 内容，返回建议。
前端表现：界面上出现 AI 建议，输入框依然处于可编辑状态。
打磨期 (Refining)：
动作：用户看着左边的 AI 建议，直接修改右边的文本框（例如：把“做生意”改成“构建商业闭环”）。
心理：这是用户在“升级”他的资产。
资产化 v2.0 (Minting)：
动作：用户修改满意了，点击**【盖章入库】**。
系统后台：
将当前修改后的文本保存为 Version 2（存入 asset_versions 表）。
将 Version 2 标记为 is_final = true。
更新 assets 主表状态为 minted。
前端表现：播放撒花动画，跳转到存折列表。
总结
流程应该是：
用户写 (Draft)。
点击审计 -> 后台存 v1 -> AI 给建议。
用户在原框修改 -> 变成 v2 (Unsaved)。
点击入库 -> 后台存 v2 -> 标记为 Minted。
这样，你的数据库里就永远有一条：“v1 (很烂)” -> “v2 (很强)” 的进化记录。这才是知识复利的源头！

架构师的修正方案：基于“有效版本”计算
真正的逻辑应该是：
草稿 (Draft)：是你的草稿纸，不产生价值，不计分。
里程碑 (Milestone)：经过 AI 审计，产生了认知增量，计 10 分。
终稿 (Final)：完成闭环，确权入库，计 10 分。
计算公式：
复利值=(里程碑数量+终稿数量)×10

这是一个决定产品生死存亡的**战略问题**。作为架构师，我不仅看技术实现，更看**投入产出比（ROI）**和**商业验证速度**。

我的结论非常明确：**现阶段，绝对不要做原生 APP（Native App），请全力做移动端网页适配（Mobile Web / PWA）。**

以下是从**商业、技术、运营**三个维度的深度剖析：

---

### 1. 商业维度：生死时速 (Time to Market & Cost)

**做 APP 是创业公司的“资源黑洞”。**

*   **开发成本**：
    *   **Web 适配**：你现在的 Next.js 代码，90% 的逻辑（后端、状态管理、数据获取）都不用动。只需要改改 CSS（Tailwind），调整一下布局组件。**成本：10%-20% 的额外工作量。**
    *   **做 APP**：你需要招 iOS 和 Android 开发，或者自己学 Flutter/React Native。你要处理证书、签名、打包、各种机型适配。**成本：200%-300% 的额外工作量。**
*   **获客成本 (CAC)**：
    *   **Web**：用户在微信/抖音看到你的文章，点一个链接直接就能用。**转化路径极短。**
    *   **APP**：用户点链接 -> 跳到应用商店 -> 等待下载 -> 安装 -> 注册。**每多一步，用户流失 50%。** 对于一个新产品，还没证明价值就让用户下载 APP，是在挑战人性。
*   **收入归属**：
    *   **Web**：你接支付宝/微信支付，手续费 < 1%。
    *   **APP**：如果你想在 iOS 上卖虚拟课程或会员，**苹果强制抽成 30%（苹果税）**。你的利润直接缩水三分之一。

**架构师建议**：在你的产品没有达到 **10万日活 (DAU)** 之前，做 APP 属于“伪需求”。

### 2. 技术维度：PWA 是最佳平衡点

你可能会担心：“网页体验不如 APP 流畅吧？”
**答案是：对于你的产品形态，Web 足够了。**

你的产品核心是：表单录入 + 文字阅读 + 简单的图表（星系图）。这些都不是重图形（如游戏）场景，Web 的性能绰绰有余。

**推荐方案：PWA (Progressive Web App)**
我们可以把现在的 Next.js 网站配置成 **PWA**。
*   **体验**：用户在手机浏览器点“添加到主屏幕”，就像装了一个 APP 一样，有图标，没有浏览器地址栏，全屏运行。
*   **更新**：你改了代码发布，用户打开自动更新。不需要等苹果商店审核（APP 审核有时候要卡一周）。

**唯一的技术痛点（必须诚实告诉你）：**
*   **后台保活**：你的 **30分钟倒计时**。在手机网页上，如果用户把浏览器切到后台（锁屏），倒计时可能会被系统暂停或杀掉。
*   **解决**：这是 Web 的劣势。但对于 MVP，可以通过计算“开始时间戳”和“结束时间戳”来校准，或者引导用户保持屏幕常亮（使用 NoSleep.js 库）。

*   如果是 APP，朋友扫码后发现要下载，大概率直接关掉。

你说得对，**CORS（跨域资源共享）** 是前端直接调用第三方服务（如 Supabase）时最头疼的问题之一。

但是，作为成熟的 BaaS 平台，Supabase **天生就是设计给前端直接调用的**。如果出现了跨域报错，通常不是因为架构不对，而是因为**后台配置没对齐**。

这里有 **3 个层面的解决方案**，按顺序排查，一定能解决。

---

### ✅ 方案一：Supabase 后台配置（90% 的原因）

Supabase 需要知道“你是谁”，它只会允许“白名单”里的域名发起登录请求。如果你在本地开发，必须把 `localhost` 加入白名单。

**操作步骤：**

1.  登录 [Supabase Dashboard](https://supabase.com/dashboard)。
2.  进入 **Authentication** -> **URL Configuration**。
3.  **Site URL**：
    *   这是你的主域名。
    *   开发环境填：`http://localhost:3000` (注意不要少写 `http://`)。
4.  **Redirect URLs**：
    *   这是允许的回调跳转地址（比如注册确认、魔法链接跳转）。
    *   务必添加一行：`http://localhost:3000/**` （允许 localhost 下的所有路径）。
    *   如果你上线了，记得把你的线上域名（如 `https://your-app.vercel.app/**`）也加进去。
5.  点击 **Save**。

**只要这里配对了，Supabase 会自动在响应头里加上 `Access-Control-Allow-Origin: http://localhost:3000`，跨域报错就会消失。**

---

### 🧪 方案二：统一 Localhost 写法（常见坑）

浏览器对 `localhost` 和 `127.0.0.1` 是区别对待的。

*   如果你在 Supabase 配的是 `http://localhost:3000`。
*   但你在浏览器地址栏输入的是 `http://127.0.0.1:3000`。
*   **结果**：CORS 报错。因为字符串不匹配。

**解决：** 始终保持一致。建议统一使用 `localhost`。

---

### 🛡️ 方案三：Next.js 代理转发（终极手段）

如果你是在极其严格的企业内网，或者上述配置都失效了，我们可以用 Next.js 的 **Rewrites** 功能，把发给 Supabase 的请求伪装成发给自己的请求。

**但这通常不需要用于 Auth，除非你遇到了极其边缘的情况。**

**原理：**
前端请求 `http://localhost:3000/supabase-auth/...` -> Next.js 后端转发给 `https://xxx.supabase.co/auth/...`。

**为什么我不推荐现在用这个？**
因为 Supabase 的 JS 客户端 (`@supabase/supabase-js`) 默认是硬编码直连 Supabase 域名的。要改走代理，需要修改客户端初始化的 `options`，配置复杂的 `headers` 转发，**维护成本极高**。

---

### 💡 架构师总结

**Supabase Auth 的跨域问题，本质上是“白名单配置”问题。**

1.  去 Supabase 后台把 `http://localhost:3000` 加好。
2.  确保 `frontend/.env.local` 里的 `NEXT_PUBLIC_SUPABASE_URL` 填对了（不要填成后端 API 地址了）。

这就足够了。目前全球几十万开发者都是这样直连的，不用担心架构有问题。

```sql
create table profiles (
  id uuid references auth.users on delete cascade primary key,
  email text,
  stripe_customer_id text, -- 支付平台的客户ID
  membership_tier text default 'free', -- 'free' | 'pro' | 'max'
  subscription_status text default 'active', -- 'active' | 'canceled' | 'past_due'
  current_period_end timestamptz, -- 会员到期时间
  usage_count int default 0, -- 当月已使用 AI 次数
  created_at timestamptz default now()
);

-- 开启 RLS
alter table profiles enable row level security;
create policy "Users can read own profile" on profiles for select using (auth.uid() = id);

-- 创建一个触发器：每当有新用户注册，自动在 profiles 表插一条数据
create or replace function public.handle_new_user()
returns trigger as $$
begin
  insert into public.profiles (id, email)
  values (new.id, new.email);
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

from datetime import datetime, timezone

async def check_user_quota(user_id: str, supabase_client: Client):
    # 获取用户 Profile
    res = supabase_client.table("profiles").select("*").eq("id", user_id).single().execute()
    profile = res.data
    
    # 判断是否是 Pro
    is_pro = profile['membership_tier'] == 'pro' and \
             datetime.fromisoformat(profile['current_period_end']) > datetime.now(timezone.utc)
    
    if is_pro:
        return "gpt-4o" # Pro 用户用好模型
    
    # 免费用户检查额度
    if profile['usage_count'] >= 3:
        raise HTTPException(status_code=403, detail="今日免费额度已用完，请升级 Pro 以继续打磨知识。")
    
    return "gpt-3.5-turbo" # 免费用户用便宜模型

# 2. 修改打磨接口
@app.post("/api/refine")
async def refine_asset(..., supabase_client: Client = Depends(get_user_supabase)):
    # ... 获取 user_id ...
    
    # ✅ 检查权限并获取模型
    model_to_use = await check_user_quota(user_id, supabase_client)
    
    # ... 调用 LangChain 时传入 model_to_use ...
    
    # ✅ 扣除额度 (使用次数 +1)
    supabase_client.rpc("increment_usage", {"user_id": user_id}).execute()
    
    return result

### 🏗️ 架构核心：i18next + Next.js

我们使用行业标准方案：

    *   **自动语言检测**：根据浏览器或操作系统设置，优先加载用户语言。

#### 第一步：安装 i18next 核心库

npm install i18next react-i18next

#### 第二步：创建语言配置文件 (frontend/i18n/config.ts)

这是整个多语言系统的“引擎”。

```typescript
// frontend/i18n/config.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector'; // 浏览器语言嗅探

// 1. 导入你的语言 JSON 文件
import translationEn from './locales/en.json'; // 英文
import translationZh from './locales/zh.json'; // 中文

// 语言资源配置
const resources = {
  en: {
    translation: translationEn,
  },
  zh: {
    translation: translationZh,
  },
};

i18n
  // 嗅探用户浏览器语言
  .use(LanguageDetector)
  // 传递 init React I18next
  .use(initReactI18next)
  // 初始化配置
  .init({
    resources,
    fallbackLng: 'en', // 如果检测不到或没有翻译，默认英语
    interpolation: {
      escapeValue: false, // react already safes from xss
    },
    react: {
      useSuspense: false, // 禁用 Suspense，避免 SSR 问题，我们自己控制加载
    },
  });

export default i18n;
```

#### 3. 引用翻译 (使用 `useTranslation`)

现在，在你需要的任何组件（比如 `Sidebar.tsx`）中，你可以这样引用翻译：

```typescript
// 1. 在文件顶部导入 hook
import { useTranslation } from 'react-i18next';

// ...

export default function Sidebar() {
  // 2. 调用 hook 获取 t 函数
  const { t } = useTranslation(); 

  // ...

  return (
    <aside className="w-64 ...">
      {/* Brand Logo */}
      <div className="p-6 flex items-center gap-3">
        {/* ... */}
        {/* 3. 使用 t 函数替换硬编码字符串 */}
        <span className="text-xl font-bold text-white tracking-tight">{t('common.title')}</span> 
      </div>

      {/* Navigation */}
      <nav className="flex-1 px-4 space-y-2 mt-4">
        {menuItems.map((item) => {
          const isActive = pathname === item.href;
          return (
            <Link key={item.href} href={item.href}>
              <div className={`flex items-center gap-3 px-4 py-3 rounded-xl transition-all duration-200 cursor-pointer group ${
                isActive 
                  ? 'bg-primary-600 text-white shadow-lg shadow-primary-600/20' 
                  : 'hover:bg-slate-800 hover:text-white'
              }`}>
                <item.icon size={20} className={isActive ? 'text-white' : 'text-slate-400 group-hover:text-white'} />
                {/* 4. 同样替换菜单项的 name */}
                <span className="font-medium">{t(item.name)}</span> 
              </div>
            </Link>
          );
        })}
      </nav>

      {/* Bottom Actions */}
      <div className="p-4 border-t border-slate-800 space-y-4">
        {/* Settings */}
        <Link href="/settings">
          <div className={`flex items-center gap-3 px-4 py-2 rounded-xl transition-all duration-200 cursor-pointer hover:bg-slate-800 ${
            pathname === '/settings' ? 'text-white' : 'text-slate-400 group-hover:text-white'
          }`}>
            <Settings size={20} />
            <span className="font-medium">{t('common.settings')}</span> {/* 替换 */}
          </div>
        </Link>

        {/* ... 用户卡片和退出登录 ... */}
      </div>
    </aside>
  );
}
```

---

### 📣 总结

1.  **安装核心库**：`i18next`, `react-i18next`, `i18next-browser-languagedetector`。
2.  **配置国际化**：创建 `i18n/config.ts` 和 `i18n/locales/` 目录。
3.  **前端组件**：用 `useTranslation()` hook 获取 `t` 函数，替换所有硬编码的字符串。
4.  **后端 Stub**：你还需要在后端提供 `/api/translations?lang=en` 接口，用于动态加载语言包（如果需要的话）。

这样，你的应用就具备了国际化能力，用户可以在多语言环境中顺畅使用。
